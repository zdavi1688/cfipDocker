# 远程 VPS Cloudflare WARP 代理自动化管理系统 - 开发与架构设计文档

本程序为多节点 Cloudflare WARP 住宅代理群组的宿主机自动化管理面板，提供轻量、秒级部署更新及高稳定的高性价比方案。为了保证系统的稳定性及长期维护性，特整理此设计与维护文档。

---

## 1. 核心功能及实现逻辑

### 功能 A: 容器热更新与平滑缩放模块
*   **功能描述**：当用户在面板中修改所需部署的节点数量后，系统能够秒级对其进行扩容或缩减，保障原有已存活节点不受断网影响。
*   **设计逻辑**：
    1.  更新内存配置后，同步更新宿主机持久化配置 `/root/cf-proxies/panel-config.json`。
    2.  动态构建最新的 `docker-compose.yml` 配置文件。
    3.  使用 `docker compose up -d --remove-orphans` 热更引擎替代传统的 `down` 重启，只对新增或删除的容器做物理变动，正常节点零等待、不重启、不掉线。
    4.  内存中的代理节点指标列表与系统同步：扩容时增量 Push 空白对象以触发探活；缩容时由后向前平滑 Slice 移除节点，保证低索引的运行节点不受改动。

### 功能 B: 容器密钥 / 身份本地持久化机制
*   **功能描述**：避免容器因重新创建、宿主机重启或 IP 刷新等重建操作而频繁向 Cloudflare 客户端服务器发起匿名账号自注册 (Reg) 申请，解决 Cloudflare 拥堵及拦截导致的 `Connection timeout` 部署失败问题。
*   **设计逻辑**：
    1.  指定宿主机统一密钥存放路径为 `/root/CF_Key/warp-{i}`。
    2.  在 Docker 编排配置中，将各子目录挂载至对应 WARP 容器内的身份工作目录 `/var/lib/cloudflare-warp`。
    3.  **权限兼容性障壁处理 (777赋权)**：由于容器内部常驻服务 `warp-svc` 有可能以自身专有的非 root 账号运行，如果宿主机目录权限为默认 root 属主的 `755`，会导致容器写入密钥发生 `Permission denied` 并导致容器闪退崩死。软件在自动生成最新配置矩阵和目录时，**强制通过 Node.js 执行宿主机和各子映射目录的高兼容性赋权 (chmod 777)**。任何新老容器不仅能够零障碍读取/写入属于自己的身份文件，还能实现秒级连接、零延迟启动。

### 功能 C: 定期巡检与智能净化过滤网络
*   **功能描述**：24小时无缝巡检节点网络状况、抓取出口地理和延迟，并基于 Scamalytics 提供免费 of 欺诈值评定。通过强制对极净 IP、达标 IP 以及被污染高危 IP (>= Max Score Allowed) 进行过滤筛选、动态重写和热清洗。
*   **设计逻辑**：内置 30 分钟间隔的自动守护任务进程。利用 `curl --socks5-hostname` 配合 socks 账户凭证对具体代理通道进行直测，对于异常、高欺诈值或发生 IP 重合 (冲突组) 的节点直接使用 `docker restart` 命令重置换脑，实现无限量自我净化能力。

### 功能 D: 宿主机公网出口 IP 自动监测与指纹浏览器列表渲染
*   **功能描述**：自动检测该 SOCKS5 代理管理系统所在远程 VPS 的真实公网 IP。避免指纹浏览器批量生成列表时硬编码假 IP 或要求用户人工查询并手动粘帖的繁杂操作。
*   **设计逻辑**：
    1.  **服务端主动探测**：后端服务启动时，内置探测器会依次向 `api.ipify.org?format=json`、`ipapi.co/json/` 及 `ifconfig.me` 发起轻量级出口 IP 嗅探。嗅探成功后，缓存至内存 `vpsPublicIP`，并在终端打印日志。
    2.  **前端双模式兜底兼容**：指纹导出页面在挂载时，首先读取 `window.location.hostname` 域名或 IP。如果检测到它是非 localhost 的有效远端地址，则预置为对应显示 IP。
    3.  同时，前端并行向服务端的 `/api/vps-ip` 接口发起请求，将服务端在 VPS 环境精准探测到的真实公网出口 IP 动态渲染至导出输入框内。该设计在本地测试/沙箱开发环境及真实 VPS 环境间均具备高鲁棒性的自动侦错与自适应填充能力。

### 功能 E: 深度守护线程诊断及调试日志持久化下载模块
*   **功能描述**：全天候不间断跟踪收集底层端口通信、容器交互及外网接口查询中的每一个细微指标及错误回执。提供一键下载以及一键清空日志的管理功能，供运维人员快速调错、诊断、发送排障文件给协同团队。
*   **设计逻辑**：
    1.  **分流持久化落盘**：引入伴生诊断器，捕获全流程中包括 curl 的绝对命令行、Scamalytics json/ipapi 原始通信报文、系统调用失败的详细堆栈（docker compose 或 restart）。
    2.  所有的诊断细节默认不仅实时投送至控制台控制面板，还将自动追加写入宿主机 `/root/cf-proxies/runtime-debug.log`。
    3.  **日志生命周期接口控制**：开放 `/api/debug-log/download` 流式下载和 `/api/debug-log/clear` 覆盖清空接口。前端设置对应的动作按钮，满足在长期重度负载运行下对宿主机日志盘空间的无损释放需求。

## 2. 系统开发变动管理原则

为了避免“前面刚完善，后面又搞乱”的缺陷，后续的所有代码迭代必须严格确保遵循以下底层铁律：

1.  **原则 1：严禁篡改已有逻辑**：新增任何逻辑或代码块都不得删除、屏蔽或篡改现有的已调试成熟的核心流程。
2.  **原则 2：功能颗粒模块化**：在可能的情况下，多考虑使用外部或清晰独立的逻辑判断分支，杜绝混淆整体流程。
3.  **原则 3：Debug 日志伴生**：每一次涉及到新状态或动作的推进，不管成功还是失败，都必须以 `addLog` 方法补充实时、详细的运行日志，让运维和终端用户可以随时追述行为原点。

## 3. 面板自适应安装机制 (修补变动归档)

### 3.1 路径自适应检测
在 `deploy.sh` 升级版本中，不再允许任何绝对项目路径硬编码 (如 `/root/cfip`)。脚本会自动获取当前执行目录 `BASH_SOURCE[0]` 的绝对物理路径作为当前工作根目录，并将：
1. 项目前端打包与运行日志 `panel.log` 直接输出在当前包的物理路径下。
2. 支持用户在任何合规的自定义目录件 (比如 `/root/cf`, `/root/cfip-panel`, `/home/custom/cf`) 进行拉取并一键运行部署，解除路径单一化绑定导致的崩溃。

### 3.2 跨系统调试兼容
对 `lsof` 或者高级查询工具不存在的新系统情况进行安全屏蔽 (即输出转向 `/dev/null`)，并利用 `ps` 查杀做二次底层兜底，防止因为命令缺失导致部署脚本中断爆出警告。
